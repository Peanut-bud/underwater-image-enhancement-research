# 流程一详细代码设计

## 1. 目标定位

流程一对应 [gemini流程.md](D:/科研/图像增强/思路/gemini流程.md) 中的“阶段一：基于物理模型的离线数据合成”。

这一阶段的目标不是做一个临时脚本，而是建立一个**正式的数据合成模块**，为后续阶段二、阶段三提供稳定的四元组监督数据：

- `input`：退化后的浑浊图 `I`
- `target`：清晰参考图 `J`
- `transmission`：透射率图 `t`
- `airlight`：环境光 `A`

输出结果必须严格兼容当前工程里的 [src/datasets/synthetic_quad_dataset.py](D:/科研/图像增强/src/datasets/synthetic_quad_dataset.py:1)。

## 2. 设计原则

### 2.1 总原则

流程一代码必须同时满足 4 个要求：

1. 参考 [gemini流程.md](D:/科研/图像增强/思路/gemini流程.md:1) 中的阶段一方法主线。
2. 参考 [gemini论文以及代码链接.md](D:/科研/图像增强/思路/gemini论文以及代码链接.md:1) 里提到的论文方法和开源代码。
3. 兼容当前项目已经固定下来的数据协议和后续训练接口。
4. 第一版先做“最小可运行正式版”，不追求一次写成最终论文级合成器。

### 2.2 论文对照原则

流程一重点借鉴以下论文：

- `AOD-Net`
  用于支撑基于物理散射模型合成监督数据的合理性。
- `UWCNN`
  用于指导“按水体退化类型/参数区间采样”，而不是无约束随机退化。
- `UIE Survey`
  用于约束数据构建、参数记录和实验复现思路。
- `DepthAnything`
  不是主理论来源，但作为后续真实深度估计后端的目标接口。

以下论文不作为流程一主参考，但会影响接口设计：

- `UColor`
  影响后续 `A/t` 监督目标设计，因此流程一必须保留清晰的 `A` 和 `t` 真值。
- `NAFNet`
  主要影响阶段二精炼子网，不主导流程一。
- `Zero-DCE`
  主要影响阶段四无监督损失，不主导流程一。

## 3. 流程一最终职责

流程一只负责下面这条链路：

`clean_source/images -> depth -> transmission -> physical degradation -> synthetic quadruplets -> manifest/splits`

具体展开为：

1. 扫描清晰参考图。
2. 为每张图估计深度图或深度代理图。
3. 根据配置采样水体参数和附加退化参数。
4. 按物理退化公式生成浑浊图。
5. 保存四元组及配套 metadata。
6. 生成 `train / val / test` 划分文件。

流程一不负责：

- 模型训练
- 指标评估
- 真实无监督微调
- 阶段二网络结构

## 4. 物理模型约束

### 4.1 核心公式

第一版统一采用：

`I = J * t + A * (1 - t)`

其中：

- `I`：退化图
- `J`：清晰图
- `A`：全局 3 通道环境光向量，shape 固定为 `[3]`
- `t`：单通道透射率图，shape 固定为 `[H, W]`

透射率由深度图生成：

`t = exp(-beta * depth)`

### 4.2 第一版不做的事

第一版暂时不引入：

- 三通道波长独立透射率
- 复杂海水光谱吸收模型
- 多次散射精细建模
- 生成式先验合成

原因是当前阶段二模型已经固定为：

- `A` 为 `[B, 3, 1, 1]`
- `t` 为 `[B, 1, H, W]`

流程一必须先服务于这个接口，而不是反过来破坏当前主干定义。

## 5. 目录与文件职责

### 5.1 入口文件

[src/prepare_synth.py](D:/科研/图像增强/src/prepare_synth.py:1)

职责：

- 读取配置
- 校验目录
- 收集清晰图列表
- 执行 split 划分
- 调用批量合成器
- 写出 manifest 和统计信息

不负责：

- 具体图像退化公式
- 深度估计细节
- 图像级参数采样逻辑

### 5.2 深度接口

[src/physics/depth_proxy.py](D:/科研/图像增强/src/physics/depth_proxy.py:1)

应改造成“深度后端接口层”。

建议结构：

- `estimate_depth_proxy(image_tensor)`
  占位版深度估计，先保留。
- `normalize_depth(depth_tensor)`
  统一深度范围到 `[0, 1]`。
- `build_depth_estimator(config)`
  根据配置返回占位深度后端或未来的真实深度后端。

第一版策略：

- 默认仍用亮度反比代理深度。
- 接口上为以后接 `DepthAnything` 预留开口。

### 5.3 透射率模块

[src/physics/transmission.py](D:/科研/图像增强/src/physics/transmission.py:1)

职责：

- 根据深度图和 `beta` 生成 `t`
- 做下限截断和范围裁剪

建议补充：

- `clamp_min_t` 配置支持
- 返回统计值接口，便于写 metadata

### 5.4 环境光模块

[src/physics/airlight.py](D:/科研/图像增强/src/physics/airlight.py:1)

职责应扩展为：

- 从配置采样 `A`
- 约束 `A` 为黄绿/蓝绿偏置区间
- 转成广播张量 `[1, 3, H, W]`

建议新增：

- `sample_airlight(generator, config)`
- `airlight_to_tensor(values, device, dtype)`

### 5.5 附加退化模块

[src/physics/degradations.py](D:/科研/图像增强/src/physics/degradations.py:1)

当前只有极简 `glare` 和 `noise`，后续应承担：

- 局部高光/过曝
- 颗粒噪声
- 可选轻度模糊

第一版建议只保留：

- `apply_glare`
- `apply_noise`

不建议第一轮就写太复杂的混合退化，否则很难校验。

### 5.6 核心合成器

[src/physics/synthesis.py](D:/科研/图像增强/src/physics/synthesis.py:1)

这是流程一的核心文件，应从“summary 工具”升级为“正式单图/批量合成逻辑”。

建议包含：

- `SynthesisSample`
  表示单样本的生成结果和参数。
- `SynthesisSummary`
  表示整个批次的统计结果。
- `sample_degradation_params(config, rng)`
  采样本张图的 `beta/A/glare/noise/...`
- `synthesize_sample(...)`
  对单张 clean 图生成四元组。
- `save_synthesis_sample(...)`
  将单张结果写入对应 split。
- `build_splits(...)`
  构建 train/val/test 列表。
- `run_synthesis_pipeline(...)`
  统一阶段一批处理入口。

## 6. 配置设计

### 6.1 `configs/data/synth_build.yaml` 应新增字段

建议按以下结构扩展：

```yaml
project:
  name: underwater_image_enhancement
  stage: stage1_synthesis

data:
  clean_source_root: "./data/clean_source"
  raw_field_root: "./data/raw_field"
  synthetic_root: "./data/synthetic"
  split_dir: "./data/splits"
  image_extensions: [".png", ".jpg", ".jpeg"]
  recursive: false
  output_image_format: ".png"

model:
  depth_backend: "placeholder_inverse_luminance"
  transmission_mode: "exp_beta_depth"

training:
  enabled: false

loss:
  enabled: false

output:
  manifest_path: "./data/raw_field/manifests/synth_build_manifest.json"
  overwrite: false

runtime:
  dry_run: false
  seed: 42

synthesis:
  image_size: [256, 256]
  samples_per_image: 3
  split_ratio:
    train: 0.7
    val: 0.2
    test: 0.1
  t_min: 0.15
  beta_range: [0.35, 1.2]
  airlight_range:
    r: [0.65, 0.9]
    g: [0.75, 0.98]
    b: [0.7, 0.95]
  depth:
    normalize: true
  glare:
    enabled: true
    probability: 0.5
    strength_range: [0.03, 0.12]
  noise:
    enabled: true
    mode: "gaussian"
    strength_range: [0.0, 0.03]
  presets:
    - name: "mild_turbid"
      beta_range: [0.35, 0.65]
      glare_probability: 0.25
      noise_strength_range: [0.0, 0.015]
    - name: "medium_turbid"
      beta_range: [0.65, 0.9]
      glare_probability: 0.5
      noise_strength_range: [0.01, 0.025]
    - name: "heavy_turbid"
      beta_range: [0.9, 1.2]
      glare_probability: 0.7
      noise_strength_range: [0.02, 0.03]
```

### 6.2 为什么要加 `presets`

这是流程一最该从 `UWCNN` 思路借鉴的地方。

目标不是完全复刻某篇论文的海水模型细节，而是先把“退化不是纯随机，而是按类型采样”这件事落到代码结构里。这样以后要继续细化成更接近 Jerlov water type，也有承接位置。

## 7. 单样本生成协议

### 7.1 输入

单张样本生成函数输入建议为：

- `source_path`
- `split`
- `sample_index`
- `config`
- `depth_estimator`
- `rng`

### 7.2 中间结果

单张样本至少保留以下中间变量：

- `target_image`
- `depth_map`
- `beta`
- `airlight`
- `transmission`
- `rough_input`
- `glare_applied`
- `noise_applied`
- `final_input`

### 7.3 输出文件

每个样本要写出：

- `synthetic/{split}/input/{stem}.png`
- `synthetic/{split}/target/{stem}.png`
- `synthetic/{split}/transmission/{stem}.png`
- `synthetic/{split}/airlight/{stem}.json`
- `synthetic/{split}/metadata/{stem}.json`

### 7.4 命名规则

建议用：

`{source_stem}-s{sample_id:02d}`

例如：

- `cross_01-s01`
- `cross_01-s02`

不要继续沿用现在这种缺乏统一语义的 smoke 命名方式。

## 8. Metadata 设计

### 8.1 `airlight/{stem}.json`

建议固定为：

```json
{
  "A": [0.82, 0.91, 0.90]
}
```

### 8.2 `metadata/{stem}.json`

建议至少包含：

```json
{
  "sample_id": "cross_01-s01",
  "split": "train",
  "source_image": "clean_source/images/cross_01.jpg",
  "target_image": "synthetic/train/target/cross_01-s01.png",
  "input_image": "synthetic/train/input/cross_01-s01.png",
  "transmission_image": "synthetic/train/transmission/cross_01-s01.png",
  "airlight_file": "synthetic/train/airlight/cross_01-s01.json",
  "beta": 0.72,
  "A": [0.82, 0.91, 0.90],
  "depth_source": "placeholder_inverse_luminance",
  "preset": "medium_turbid",
  "noise": {
    "enabled": true,
    "mode": "gaussian",
    "strength": 0.018
  },
  "glare": {
    "enabled": true,
    "strength": 0.07
  },
  "transmission_min": 0.19,
  "transmission_max": 0.96,
  "image_size": [256, 256],
  "seed": 42
}
```

### 8.3 为什么 metadata 要做这么全

因为后面你会需要它来做：

- 训练结果复现
- 失败样本回溯
- 参数分布检查
- 比较不同 preset 的训练收益

这是 `UIE Survey` 一类综述最强调的实验可复现性基础。

## 9. Manifest 与 split 设计

### 9.1 `data/splits/*.txt`

继续沿用一行一个相对路径。

例如：


### 9.2 总 manifest

`manifest.json` 建议包含：

- 本次运行时间
- 使用配置快照
- 输入 clean 图数量
- 每个 split 生成数量
- 每个 preset 采样数量
- 实际输出根目录

## 11. 实现顺序

建议按下面顺序写，不要同时铺太开：

1. 先扩配置文件
2. 再实现单图合成函数
3. 再实现批量落盘
4. 再实现 split 和 manifest
5. 最后补测试

## 12. 第一版测试要求

流程一至少新增这些测试：

- `test_synthesis_single_sample`
  验证单图合成输出字段齐全。
- `test_synthesis_save_layout`
  验证落盘目录和文件名符合协议。
- `test_prepare_synth_smoke`
  验证入口能用小样本生成一批四元组。
- `test_manifest_generation`
  验证 manifest 和 splits 写出成功。

## 13. 当前结论

流程一后续实现时，应坚持下面这句话：

**按论文方法约束接口，按工程需求约束输出，按最小可运行原则逐步增加物理复杂度。**

换句话说，第一版流程一不是为了“物理最完美”，而是为了：

- 合理
- 可复现
- 可扩展
- 能直接服务阶段二和阶段三

后续真正开始写代码时，优先顺序应为：

1. 先把阶段一从“校验脚手架”改成“可落盘合成器”
2. 再把深度后端接口化
3. 再把 `preset` 采样引入配置
4. 最后再继续逼近论文里的更复杂真实退化
